# 🏢 Engawa Cycle - Master Blueprint v4.0
> **Minimal LLM Workspace の次なる進化層。**
> 「美しいUIとシンプルなコード（Minimalism）」の哲学を完全に維持したまま、バックエンドに強力な「自律型AI組織（Company OS）」を構築するための正典（Source of Truth）。

---

## 1. コア・フィロソフィー（Core Philosophy）
本プロジェクトを拡張する開発AIは、以下の原則を絶対に遵守すること。

1. **極小のUI / 極大の知能 (Minimal UI, Maximal Intel)**:
   - 複雑な推論やAI同士の議論はすべてバックエンド（裏側）で行う。人間（CEO）のダッシュボードは、究極にシンプルでノイズのないAppleライクなデザインを維持する。
2. **証拠主義 (Evidence-Based)**:
   - AIは提案する際、必ずソース（URLやデータ）を提示する。
3. **人間絶対優位の原則 (Human-in-the-loop Protocol)**:
   - 資金の移動、重要なメールの送信、インフラの変更など、不可逆な操作は **100%人間の最終クリック（Approve）** を必須とする。

---

## 2. 組織アーキテクチャ（AI社員名簿）
単一の巨大なプロンプトを捨て、役割を分割（Separation of Concerns）した専門特化型AIを構築する。

| 役職 (Role) | 責務 (Responsibility) | 保有ツール (Tools) | 実装モデル（推奨） |
|---|---|---|---|
| **🏢 CEO (Human)** | 方向性の決定、予算設定、最終承認。 | UIダッシュボード全般 | 大城 様 |
| **🧠 Manager AI** | 進行管理。CEOからの指示を分割し、下位AIにタスクを割り振る。 | Task Tracker, DB Read/Write | GPT-4o / Claude 3.5 Sonnet |
| **🔍 Researcher AI** | 指定されたテーマの深掘り、市況監視、関連ニュースの収集と要約。 | Web Search, Scrape, News API | Gemini 2.5 Flash / Pro |
| **📈 Trader AI** | 収集データに基づく価格予測、トレードの実行案の策定。 | Crypto API (Read-only) | GPT-4o (論理的推論用) |

---

## 3. 次期開発スケジュール（Phases 5-8）
複雑化を防ぐため、1つのコードにすべてを詰め込まず、以下のスケジュールに従って段階的に機能（モジュール）を実装していく。

### 🕒 Phase 5: 手足の拡張（Function Calling 基礎実装）
* **目的**: AIが外部世界（API）にアクセスできる基盤を作る。
* **開発内容**:
  1. `src/lib/tools/` ディレクトリの新設。
  2. 最初は安全な「GetWeather (天気取得)」や「GetCryptoPrice (任意トークンの価格取得)」ツールを実装。
  3. AIがプロンプト内でこれらのツールを自ら判断して呼び出すバックエンド（`route.ts` の進化）の構築。
* **完了条件**: チャットで「今日のBTCの価格は？」と聞き、AIがリアルタイムAPIを叩いて正確な最新価格を回答できること。

### 🕒 Phase 6: 思考のループ（Agentic ReAct 導入）
* **目的**: 「一度聞いて一回答える」から、「タスクが終わるまで一人で考え続ける」非同期基盤の構築。
* **開発内容**:
  1. Vercel AI SDK (または自作の軽量ReActループ) の導入。
  2. **Thought (思考) → Action (ツール実行) → Observation (結果確認)** のプロセスを構築。
* **完了条件**: 「BTCとETHの価格を比較し、ニュースも読んで結論を出せ」という複雑なタスクに対し、裏側で複数回のツール実行を行ってから最終回答を導き出せること。

### 🕒 Phase 7: エンガワ・オフィス（マルチAI・オーケストレーション）
* **目的**: 複数のAI社員を雇用し、連携させる。
* **開発内容**:
  1. データベース（Supabase等）に `agents_registry` と `tasks_queue` を作成。
  2. CEOが入力した指示を Manager AI が受け取り、タスクを分割して Researcher AI 等にキュー（待ち行列）として渡す仕組みの構築。
* **完了条件**: 複数のAIが連携して1つのレポートを完成させ、ダッシュボードに「社長、分析が完了しました」と自発的に報告してくる非同期通知システムの完成。

### 🕒 Phase 8: ビジュアル・ポリッシュ（Minimalism UI/UX）
* **目的**: 裏側の複雑なAIの働きを、美しく直感的なUIで表現する。
* **開発内容**:
  1. ダッシュボード上に「🤖 Researcher がWebを検索中...」「🧠 Manager が結果を取りまとめ中...」といった、AIの現在の思考状態（ステータス）を表現する洗練されたマイクロアニメーションの追加。
  2. 承認待ちタスクの [Approve] / [Reject] ボタンの洗練化。
* **完了条件**: AppleのOSのように、ユーザーが裏側の複雑さを一切感じさせない、極限まで磨き上げられたユーザー体験の提供。

---

> [!WARNING]
> **次期開発AIへの厳命事項**
> この青写真を読んで開発を再開するAIは、**「コードの肥大化」を絶対に避けること**。
> もし既存の `page.tsx` や `route.ts` が 300 行を超えるような場合は、必ず「コンポーネントの分割」や「モジュール化」を提案・実行し、可読性と美しさを維持しなければならない。
